# データ可視化設計仕様(Prometheus メトリクス契約 + Grafana ダッシュボード設計ガイド)

> GUI を持たないため、テンプレートの「画面のチャート」は、_inference.md の方針に従い
> **/metrics の Prometheus メトリクス契約**と**外部監視基盤(Grafana 等)でのダッシュボード設計ガイド**として翻案する。
> **しきい値判定・アラート発報は外部監視基盤の責務**であり、本体はメトリクス契約の安定提供に徹する(SP-005、CTP-009)。

## Prometheus メトリクス契約

情報.tsv「メトリクス」の属性(Topic別最終収集時刻 / 処理件数 / 配信失敗数 / DLQ件数 / 滞留数)をそのまま公開する。これ以外のメトリクスは RDRA に無いため契約に含めない。

### メトリクス一覧(名称は候補。実装時に確定し、確定後は後方互換を維持する)

| メトリクス名候補 | 型 | ラベル | 内容(情報「メトリクス」の属性) |
|----------------|----|-------|------------------------------|
| `file_pubsub_last_collect_timestamp_seconds` | gauge | `topic` | Topic 別最終収集時刻(Unix 秒)。収集成功のたびに更新 |
| `file_pubsub_processed_total` | counter | `topic` | 処理件数(収集〜配信サイクルで処理したメッセージ累計) |
| `file_pubsub_delivery_failure_total` | counter | `topic` | 配信失敗数(Subscription 配信の失敗累計。リトライによる再失敗を含む) |
| `file_pubsub_dlq_count` | gauge | `topic` | DLQ 件数(現在 DLQ に隔離されているメッセージ数) |
| `file_pubsub_backlog_count` | gauge | `topic` | 滞留数(収集済みでまだ全 Subscription へ配信完了していないメッセージ数) |

### 契約上の注意

| 項目 | 規約 | 根拠 |
|------|------|------|
| ラベル | ラベルは `topic` のみ。Topic ごとに収集ソースや Subscription 構成が異なるため、異常検知の単位は Topic とする | 情報「メトリクス」(関連情報: Topic)、UC: 外部監視基盤でTopic別メトリクスを観測する |
| 集計方式 | メトリクスはインメモリ集計で永続化しない。**デーモン再起動で counter はリセットされる**ため、監視側は `rate()` / `increase()` ベースで扱い、生値の単調増加を前提にしない(蓄積は外部監視基盤側の責務) | arch-design.yaml storage_mapping E-012(cache)、LR-302 |
| 公開方法 | `GET /metrics`(Prometheus テキスト形式)。ポートは設定 YAML の `metrics_port` | SP-005、情報「設定」 |
| 死活 | `GET /healthz` がデーモン稼働中に 200 を返す。メトリクスとは役割を分け、死活はこちらで判定する | UC: /healthzと/metricsをHTTPで公開する |
| アクセス特性 | HTTP アクセスは監視ポーリングのみを想定(オンライン応答系なし)。スクレイプ間隔はポーリング間隔と独立に設定してよい | CTP-008、SP-005 |
| 冗長化(lease)メトリクス | active/standby 冗長化(REQ-015)を導入しても、lease 状態・active 昇格回数等の専用メトリクスは**契約に追加しない(見送り)**。RDRA 情報「メトリクス」の属性に無く、active 昇格・standby 待機・降格・lease 奪失は構造化ログ(event_type)で十分観測でき、`/healthz`(active のみ 200)で稼働は判定できるため。RDRA に無いメトリクスを発明しない方針(本表冒頭)を維持する | 情報「メトリクス」、SPEC-015-03、spec-decision-009 |

## 可視化対象(ダッシュボードのパネル一覧)

外部監視基盤(Grafana 等)で構成するパネルの設計指針。データ可視化ルール(観点別チャート選定)に従う。

| パネル | 指標/データ | 観点 | 推奨チャート |
|-------|-----------|------|------------|
| 死活 | `/healthz` 監視結果(Prometheus の `up` 等、監視基盤側の取得方式に従う) | 状態 | ステータスパネル(UP/DOWN の単一値) |
| 収集の鮮度 | `time() - file_pubsub_last_collect_timestamp_seconds`(topic 別の最終収集からの経過時間) | Comparison(Topic 間比較) | 棒グラフ または テーブル(経過時間の大きい順) |
| 処理量の推移 | `increase(file_pubsub_processed_total[1h])` 等の topic 別処理件数 | Trend(傾向) | 折れ線グラフ / エリアチャート |
| 配信失敗の推移 | `increase(file_pubsub_delivery_failure_total[1h])` | Trend(傾向) | 折れ線グラフ |
| DLQ 滞留 | `file_pubsub_dlq_count` | Comparison(Topic 間比較)+ 状態 | 単一値(合計)+ topic 別棒グラフ |
| 滞留(backlog) | `file_pubsub_backlog_count` | Trend + Comparison | 折れ線グラフ(topic 別) |

## Grafana ダッシュボード設計ガイド

### 情報の階層化(全体サマリー → ドリルダウン)

認知負荷の原則(一度に表示する指標は 4〜5 個まで)に従い、2 階層で設計する。

- **最上段(全体サマリー、4 パネル)**: ①死活(UP/DOWN) ②DLQ 件数合計 ③滞留数合計 ④直近の配信失敗数合計。運用者が「いま対処が必要か」を 10 秒で判断できる構成にする。
- **ドリルダウン(Topic 別異常検知)**: topic ラベルで絞り込める topic 別パネル群(収集鮮度・処理量・失敗・DLQ・滞留)。Grafana の variable(`topic`)でフィルターする。
- **詳細データへの遷移**: ダッシュボードは「どの topic が異常か」までを示し、message_id 単位の特定は `status` コマンドと構造化ログへ誘導する(ダッシュボードのパネル説明文に手順を記載する)。

### データストーリーテリング

- **ナラティブ**: 「全 Topic のファイルが滞りなく流れているか。止まっているなら、どの Topic のどの段階(収集 / 配信 / DLQ)か」。
- **比較軸**: 単一値では判断できないため、①Topic 間比較(どの Topic だけ異常か)②過去比較(処理量が通常時間帯と比べて急減していないか)を基本とする。
- **アクション(後工程からの逆算)**: 各パネルは見た後のアクションに対応させる — 死活 DOWN → 再起動(冪等再開)、収集鮮度の停滞 → Producer/収集ソース疎通の調査、配信失敗増 → 構造化ログ調査、DLQ > 0 → `status` で隔離理由を確認し再送(`replay`)/破棄を判断。

### 認知負荷への配慮

- ワーキングメモリの限界(4〜5 項目)を考慮し、サマリー段は 4 パネルに制限する。
- Data-Ink Ratio: 背景色・3D 効果・不要なグリッド線を使わず、しきい値色(正常/異常)のみ意味を持たせる。
- ゲシュタルトの法則: 「収集系(鮮度・処理量)」と「配信系(失敗・DLQ・滞留)」を行でグルーピングし(近接)、同一 topic は同色で揃える(類同)。

## 推奨アラートルール例(外部監視基盤側で定義)

以下は外部監視基盤(Prometheus Alertmanager / Grafana Alerting 等)に設定するルールの**例**である。しきい値の決定・判定・発報はすべて外部監視基盤の責務であり、file-pubsub 本体はしきい値を持たない(SP-005)。しきい値は Topic のポーリング間隔・業務上の許容遅延に合わせて導入先が調整する。

| アラート | PromQL 例 | 意図 |
|---------|----------|------|
| 死活異常 | `up{job="file-pubsub"} == 0`(または /healthz の probe 失敗) | デーモン停止の検知 → 再起動(冪等再開)につなげる |
| 最終収集時刻の停滞 | `time() - file_pubsub_last_collect_timestamp_seconds > <許容秒数>` | 収集が止まっている Topic の検知(Producer 停止・収集ソース疎通障害等) |
| DLQ 滞留 | `file_pubsub_dlq_count > 0` | 恒久的な配信失敗の発生 → 運用者の対処判断(再送/破棄)を促す |
| 配信失敗の増加 | `increase(file_pubsub_delivery_failure_total[15m]) > <許容件数>` | 一時障害の多発検知(リトライで回復中でも傾向を把握する) |
| 滞留の増加 | `file_pubsub_backlog_count > <許容件数>` | 配信が処理量に追いついていない状態の検知 |

## 責務境界の明記

| 責務 | file-pubsub 本体 | 外部監視基盤(Prometheus / Grafana 等) |
|------|:---------------:|:------------------------------------:|
| メトリクスの算出・公開(/metrics) | ○ | - |
| 死活エンドポイント(/healthz)の提供 | ○ | - |
| メトリクスの時系列蓄積 | -(インメモリのみ・再起動でリセット) | ○ |
| しきい値判定 | - | ○ |
| アラート発報・通知 | - | ○ |
| ダッシュボード表示 | - | ○ |
| 障害対応・再送判断(人) | -(配信基盤運用者が status / ログ / replay で実施) | -(検知を運用者へつなぐまで) |

24 時間の自動監視は外部監視基盤が担い、人による対応(障害調査・再送判断)は営業時間内の配信基盤運用者が行う(CTP-009)。
