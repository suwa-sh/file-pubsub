変更要望(Codex レビュー指摘の取り込み): 冪等性・履歴保全のハードニング 2 件。

① message_id 一意化(REQ-007 / SPEC-007-01 精緻化):
  message_id は収集時刻(秒精度)+ Topic + 元ファイル名で採番するが、同一秒に同名ファイルを複数回収集すると
  message_id が衝突し、先行メッセージの Archive・Manifest を上書きして履歴を失う。特に push 受信モード(inbox)の
  rename/marker は安定待ちなしで即収集するため、短時間の同名再 put で同一秒衝突が起きやすい。
  「履歴を失わない」保証を満たすため、同一秒の同名再収集では連番付与で message_id を一意化する。

② marker + copy の残存マーカー(REQ-014 / SPEC-014-02 精緻化):
  完了検知=marker・残す(copy)では取り込み後もマーカーを残置し処理済み管理で識別する。しかし同名で内容の異なる
  本体を再 put すると、残存する旧マーカー(処理済み)が新本体の完了契機として誤用され、producer が新しいマーカーを
  置く前に書き込み途中の新本体を取り込むおそれがある。既に処理済みのマーカーは新たな完了契機にしない(新マーカー待ち)。

影響 UC: ファイルを収集する(Collect)。後段(Archive / Fan-out / Manifest)は不変。差分更新で反映する。
